Introduction

Software Layers

Typically we want to structure applications into layers, in order to keep them understandable. As an example, we might have a web application with the following structure:

web API
------------------------------
business logic
------------------------------
persistence
  and upstream service clients

strict digraph layers {
    shape=box;
    a[label="API"];
    b[label="business logic"];
    c[label="persistence and upstream service clients"];
}

where the flow of control is from top to bottom, and then back up again. I.e., this application receives requests through API endpoints exposed via HTTP; the request data is validated and mapped to events in the business domain, which in turn can trigger access to the application database or calls to other services, these then return data back to the business logic; and finally, we return a response for the web API request.

This is not at all a new idea.

Layer Violation

The important idea behind the diagram above, is that each layer can only interact with the layer below; and only trough a well defined interface. However, even with the best intentions, an application code base which is set up to follow this architecture can quickly evolve into a tangled mess where business logic is implemented in HTTP handlers, and calls are made directly to supporting services.

This degrades the understandability of the implementation over time, makes it harder to adapt to new domain requirements, and frustates new consributors.

Haskell Packages

To prevent the layer separation from being broken too easily we can try to enlist the help of language and build system features.

Consider a situation where we have at least one module per layer. A module in a layer can access functions in modules in the layer below, by importing these modules. We can only prevent access to functions in layers further below by just not importing modules from those layers.

But once each layer contains more than a few modules, it may no longer be apparent which modules belong to which layers, and which modules are allowed to import which other modules. Naming module conventions may be able to help a bit, but they still don’t provide automatically enforced boundaries. Unless there is an automatic check to flag import relations that violate the layer separation, sooner or later someone will write code that does.

So we need a way to formally describe which modules are part of which layer, and have that checked at build time. One obvious way to do that in Haskell is using cabal packages and package components. That is, we put the modules for the layers into separate packages.

As a starting point, we put all modules into the main library of a layer package. Then we can express the dependency of an upper layer on a lower layer by adding the lower layer package library as an item to the build-depends: field in the upper layer package library. Since we do no add layer packages from further below, the upper layer can not see or access layers below and therefore not skip layers. Circular dependencies of packages are likewise not allowed, so we can also be sure that nowhere in a lower layer is there a call to any function from a layer above.

To actually pin down how a layer interacts with the layer below it, we also need to make sure that a layer only calls into a layer below via a defined interface. We might try to enforce this by only listing modules that implement the public interface to a layer as exposed modules in the respective package component description, and keep all other modules private. This certainly works for preventing the layer above from seeing anything but the public interface.

But if we want to implement a test component in the package, we need to expose all the modules whose functions we want to test.

Package Pattern for Layer Isolation

the solution is to separate out the interface specification in the lower layer from the implementation into a third package, that is depended upon by both the package for the upper layer and the implementation package for the lower layer. the interface specification could be in the form of type aliases for functions, and associated parameter types; or better yet, in the form of an algebraic effect type.

we get the following picture

   upper layer package -------\
              |
              V
           lower layer interface package
              A
              |
lower layer implementation -------/
package

the upper layer can now call into the lower layer by using higher order function parameters, or effects. this requires a further package to actually supply the implementation in the lower layer to the consumers in the upper layer by applying consuming functions in the upper layer to implementation in the lower layer, or else, by invoking effect runnners from the lower layer. so we get this picture:

         upper layer package -------\
           A                        |
           |                        V
integrating package -------> lower layer interface package
           |                        A
           V                        |
      lower layer implementation -------/
      package

the horizontal dependency may not actually be needed.

with these package dependency relations in place, we will get a build error if upper layer modules try to import modules from further down the stack. great!

the integrating package can see all modules, and can connect the pieces.

there will be packages that fall outside the layer hierarchy, e.g., the integrating package and packges containing value types (both as part of a project, and also from the wider ecosystem). so we should establish a naming convention for packages and libraries to make it clear which of them are part of the layer hierarchy and which role they play there.

Automatically Enforcing Layer Boundaries

An Example

The project file

packages: *

The API package

<<cabal-header>>
name: foo-api

library
  exposed-modules: Api
  build-depends: base >= 4.0 && < 5.0, foo-core
  default-language: GHC2021

module Api where

import Core qualified as Core
import System

endpoint :: [String] -> IO ()
endpoint = \case
  ["show"] = do
    (Core.Name name, Core.Age age) <- Core.getPerson
    putStrLn $ "Name: " <> name <> ", " <> "Age: " <> (show age)
  ["celebrate"] = Core.incrementAge
  ["create", name] = Core.createPerson name
  _ = System.exit 1

The core package

<<cabal-header>>
name: foo-core

library
  exposed-modules: Core
  build-depends: base >= 4.0 && < 5.0, foo-persistence
  default-language: GHC2021

module Core where

import Persistence qualified as Persistence

data Person = Person { name: Name, age: Age }

newtype Name = Name String

newtype Age = Age Integer

-- setName :: GetRecord -> SetRecord -> Name -> IO ()
-- setName getRecord setRecord name = do
--   (_name, age) <- getRecord
--   setRecord (name, age)

getPerson :: GetRecord -> IO Person
getPerson getRecord = do
  (Persistence.Name name, Persistence.Age age) <- getRecord
  pure $ Person (Name name) (Age age)

incrementAge :: GetRecord -> PutRecord -> IO ()
incrementAge getRecord putRecord = do
  (Persistence.Name name, Persistence.Age age) <- getRecord
  putRecord (Persistence.Name name, Persistence.Age $ age + 1)

createPerson :: PutRecord -> Name -> IO ()
createPerson putRecord name = do
  putRecord (Persistence.Name name, Persistence.Age 0)

The persistence package

<<cabal-header>>
name: foo-persistence

library
  exposed-modules: Persistence
  build-depends: base >= 4.0 && < 5.0, extra
  default-language: GHC2021

module Persistence where

import Data.List.Extra

newtype Name = Name String

newtype Age = Age Integer

type GetRecord = FilePath -> IO (Name, Age)
getRecord :: GetRecord
getRecord fp = do
  lines <- lines <$> readFile fp
  case words <$> lines of
    [["Name:", name],["Age:", age]] -> pure (Name $ trim name, Age $ read age)

type PutRecord = FilePath -> Name -> Age -> IO ()
putRecord :: PutRecord
putRecord fp (Name name) (Age age) = do
  writeFile fp $ "Name: " <> name <> "Age: " <> (show age)
